Lotus defines its visual language through theme tokens for color, typography,
spacing, and shape.

## Tailwind and Lotus tokens

Tailwind handles page layout, spacing, typography utilities, and responsive
behavior. Lotus tokens define the semantic layer on top of that utility system:
surface colors, foreground colors, interactive accents, prose styling, fonts,
and shared radius values.

This split keeps authoring practical. Tailwind classes describe structure and
composition, while `--lotus-*` tokens preserve a consistent visual language
across layouts and MDX components.

The `appearance.gray` setting renders as an attribute such as
`data-gray="neutral"`. Lotus maps that selector to `--lotus-gray-*` aliases, then
derives `--lotus-text`, `--lotus-text-strong`, `--lotus-surface`, code
backgrounds, borders, and subtle dividers from that scale. The theme uses
`@theme static` to keep supported Tailwind color variables in the generated CSS.

## Why Lotus uses semantic color tokens

Lotus ships semantic tokens such as `--lotus-background`, `--lotus-text-strong`,
`--lotus-text`, and `--lotus-accent` so components can respond to theme changes
without rewriting every individual utility class. The default palette uses
`oklch(...)` values because they are perceptually uniform and easier to tune
across light and dark surfaces than raw `r g b` channel triplets.

In practice, `oklch(...)` makes it easier to adjust contrast, chroma, and tone
with predictable results while still exposing a semantic token API to the rest
of the theme.

## Radius and theme attributes

The page shell reads `appearance.defaultTheme` and `appearance.radius` from
`src/theme.config.ts` and exposes them as document attributes. A system default
renders as `data-theme="system"` so CSS can follow the visitor's
`prefers-color-scheme` setting, while explicit light and dark defaults render
the matching fixed theme.

Lotus components consume the matching radius tokens so cards, callouts, and
other surfaces stay aligned with the selected scale.

```ts
appearance: {
  defaultTheme: 'system',
  radius: 'medium',
}
```

## Package component bridge

The shared package components remain package-owned and read `--pl-*` variables.
Lotus maps those bridge variables to the semantic theme layer in
the bundled theme tokens, including `--pl-text`, `--pl-background`,
`--pl-accent`, `--pl-accent-soft`, `--pl-accent-contrast`, and the radius
tokens. This keeps package components portable while letting Lotus own the
release palette.

## Core tokens

| Token | Purpose |
|---|---|
| `--lotus-gray-50` ... `--lotus-gray-950` | Alias for the configured Tailwind neutral scale |
| `--lotus-background` | Page background |
| `--lotus-surface` | Subtle panels and neutral surfaces |
| `--lotus-code-background` | Code block and inline code background |
| `--lotus-code-border` | Code block border |
| `--lotus-text-strong` | Headings, section labels, active text |
| `--lotus-text` | Main body text |
| `--lotus-text-muted` | Secondary text and inactive navigation |
| `--lotus-text-subtle` | Low-emphasis affordances, borders, separators |
| `--lotus-accent` | Primary interactive color |
| `--lotus-accent-soft` | Muted active and hover backgrounds |
| `--lotus-accent-contrast` | Text on accent backgrounds |
| `--lotus-radius-md` | Medium component radius |
| `--lotus-radius-full` | Fully rounded controls and badges |
| `--lotus-docs-chrome-height` | Combined header and subnav height for sticky offsets |
